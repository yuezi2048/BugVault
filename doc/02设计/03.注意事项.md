

### 🏁 开工前最后确认的 4 个代码细节（避坑指南）

#### 1. 核心大坑：Embedding 模型的“单例模式”与初始化时机

- **问题：** 无论是调用 API 还是本地跑轻量级向量模型（如 `FastEmbed`），模型加载或连接池建立都是非常耗时的。
- **避坑做法：** **绝对不要**在每次 Claude 调用 `retrieve_bug_experience` 工具时去实例化 Embedding 模型。必须在 `main.py` 启动 MCP Server 时，就完成 `LanceDBClient` 和 Embedding 模型的初始化，并作为单例（Singleton）通过依赖注入传给 `services` 层。

Python

```
# 正确做法伪代码 (main.py)
db_client = LanceDBClient(db_path="~/.bugvault") 
await db_client.initialize() # 提前拉起向量模型并热身

# 将准备好的实例挂载到 MCP Tool
@mcp.tool()
async def retrieve_bug_experience(query: str):
    return await retrieval_svc.search(query, db_client=db_client)
```

#### 2. 工具调用的异常处理：永远不要抛出异常 (`raise Exception`)

- **问题：** MCP 是 Claude 与你通信的桥梁。如果在你的 `services` 层里因为 Pydantic 校验失败或者数据库超时而写了 `raise ValueError(...)`，这会导致 MCP 工具调用直接崩溃（Crash），Claude 会认为工具坏了，从而中断排障。
- **避坑做法：** 在 `mcp_tools/` 的入口处，必须加上大大的 `try...except`。如果发生任何业务错误，**把错误信息包装成自然语言的字符串返回给 Claude**。

```python
# 必须这样写
try:
    # 你的核心逻辑
except MissingFieldError as e:
    # 返回字符串，触发大模型自我反思
    return f"❌ 记录失败：缺少必填项【{e.field}】。请向用户询问该信息后，再次调用本工具。"
except Exception as e:
    return f"⚠️ 系统内部异常：{str(e)}。请尝试简化查询参数或通知开发者。"
```

#### 3. 阻塞函数的异步化隔离 (Asyncio 护城河)

- **问题：** LanceDB 的某些本地 I/O 操作（或者你用来写 Markdown 备份的 `open(file, 'w')`）是同步（Sync）阻塞的。如果在 `async def` 里面直接调用同步函数，会卡死 MCP 的事件循环，导致 Claude 等待超时。
- **避坑做法：** 在 `database/` 层遇到磁盘读写或重的 CPU 计算时，习惯性地使用 `asyncio.to_thread` 进行包装，把脏活踢给后台线程。

```python
import asyncio

async def insert_record(self, record: BugRecord):
    # 将同步的 lancedb 插入操作放入线程池执行
    await asyncio.to_thread(self.table.add, [record.model_dump()])
```

#### 4. 本地数据库的默认路径设定

- **问题：** 由于这是一个纯本地工具，数据库和 Markdown 备份存在哪里很关键。如果写死相对路径（如 `./data`），用户在不同目录下启动 Claude 时，会生成一堆零散的数据库碎片。
- **避坑做法：** 在 `config.py` 中，强制将数据根目录设定为用户的 `Home` 目录（例如 `~/.bugvault/`），这样无论在哪里调起 MCP 服务，全局的 Bug 经验都汇聚在同一个库里。
