## 1. RAG 评估 → 策略模式（Strategy Pattern）

**改动**：单一 `RAGEvaluator.evaluate_sync()` → `RAGEvalStrategy` Protocol + `SimpleRAGEvalStrategy` + `ClaimLevelRAGEvalStrategy` + `RAGEvaluator` Facade

| 考量点 | 决策依据 |
|---|---|
| **开闭原则** | 后续加新策略只需新增类实现 Protocol，不改现有代码 |
| **评估粒度可选** | `eval_depth` 参数让 Agent 运行时选择：simple（~300 tokens）适合快速评估，claim_level（~1500 tokens）适合深度验证 |
| **职责分离** | Facade 负责路由和熔断，策略类只做一件事：评估 |

## 2. Claim-Level 评估 → 三步骤 CoT

**改动**：新增 `ClaimLevelRAGEvalStrategy`，强制 LLM 执行声明提取 → 逐条验证 → 打分

**流程**：
```
query + docs → Step 1: 提取原子事实声明
            → Step 2: 逐条判断 supported/unsupported/partial
            → Step 3: faithfulness = supported / total, context_relevance (0-5)
```

| 考量点 | 决策依据 |
|---|---|
| **可审计** | 每条 claim 的 `Supported` + `Reason` 可追溯，不是黑箱打分 |
| **faithfulness ≠ relevance** | 两个维度解耦，可能出现 `context_relevance=2.0`（检索方向偏）但 `faithfulness=1.0`（没幻觉） |
| **输出格式** | `claims_analysis: [{claim, supported, reason}]` + `suggested_action: CONFIDENT/PARTIAL/CAUTION/INSUFFICIENT` |

## 3. 双重降级保护（Double Fallback）

**改动**：`_claim_level_or_fallback` 中先检查配额，再用 try/except 包裹 evaluate()

**流程**：
```
claim_level 请求
  → 配额超限? → 降级 simple, strategy_used="simple"
  → try evaluate() → 异常? → 降级 simple, strategy_used="simple (fallback_from_error)"
```

| 考量点 | 决策依据 |
|---|---|
| **小模型不可靠** | deepseek-v4-flash 在复杂 CoT 下频繁输出残缺 JSON |
| **绝不阻塞主链** | 评估是增值功能，不能因为评估崩溃导致检索失败 |
| **可见性** | `strategy_used` 字段让 Agent 知道实际执行了降级 |

## 4. Vector + FTS 双路召回 → RRF 融合

**改动**：纯 ANN → 并行 Vector ANN + FTS BM25，RRF(k=60) 融合

| 考量点 | 决策依据 |
|---|---|
| **量纲不可加** | 余弦距离（[0,2]）和 BM25 分数（对数词频，无上限）量纲不同，直接加权求和无意义 |
| **RRF 只用排名** | `score = Σ 1/(60 + rank_i)` 只依赖相对顺序，天然跨路兼容 |
| **K=60** | 业界标准值（RRF 原始论文推荐），对 5-20 条和 ~200 条结果集均稳定 |

**已否决替代方案**：Score Fusion（归一化后加权求和）——归一化方法对异常值敏感，不如 RRF 鲁棒。

**双表验证**：
```
bug_records: 初始 119 条 → 导入后 ~2108 条
bugvault_chunks: 初始 238 条 → 导入后 ~4200 条
FTS 索引: 双表均已构建
```

## 5. Cross-Encoder 精排（弃用 ColBERT）

**改动**：RRF → `rerank()` 时域调整 → Cross-Encoder 二次打分 → Top-K 截断

| 考量点 | 决策依据 |
|---|---|
| **候选集规模** | 仅 5-20 条，Cross-Encoder O(n) 开销可忽略 |
| **ColBERT 的代价** | 需要独立 PyTorch 索引（~1.5GB），LanceDB 无法复用 |
| **Cross-Encoder 收益** | 80MB ONNX 模型，零新依赖，二次交叉注意力显著提升排序质量 |
| **模型默认** | `Xenova/ms-marco-MiniLM-L-6-v2`（80MB，英文，CPU），可选 `BAAI/bge-reranker-base`（中文） |

详见 [ADR: Cross-Encoder vs ColBERT](docs/refer/设计/adr-cross-encoder-vs-colbert.md)

## 6. 元数据预过滤（Metadata Pre-filtering）

**改动**：`_RETRIEVE_SCHEMA` 新增 `target_tech_stack` + `target_project_name`，检索前 `WHERE LOWER(tech_stack) LIKE '%python%'`

| 考量点 | 决策依据 |
|---|---|
| **跨语言混淆** | 纯语义检索会把 Python `ModuleNotFoundError` 和 Java `ClassNotFoundException` 混为一谈 |
| **列式过滤代价低** | LanceDB 列式存储，WHERE 过滤在向量计算前缩小候选集，时间 < 1ms |
| **大小写容错** | `LOWER()` 确保 `Python` / `python` / `PYTHON` 均匹配 |
| **SQL 注入防护** | `_sanitise_filter_value()` 只允许 `[a-zA-Z0-9_\-\. ]` |
| **双路透传** | WHERE 条件同时作用于 Vector ANN 和 FTS BM25，不出现一路过滤另一路全表扫描 |

## 7. Small-to-Big RAG（Chunk 为主检索入口）

**改动**：检索链路从 `search(bug_records)` 改为 `search_chunks(bugvault_chunks)` → `parent_id` 聚合 → `fetch_records_by_ids(IN ...)` → Cross-Encoder

**漏斗顺序**（严禁越级）：
```
Query → chunks 双路召回 → RRF → parent_id 去重 → IN 批量回捞 → Cross-Encoder → Top-K
```

| 考量点 | 决策依据 |
|---|---|
| **短文本聚焦度** | 800 字 chunk 比数千字 parent 的语义更集中，ANN 召回精度更高 |
| **IN 防 N+1** | `fetch_records_by_ids` 用 `WHERE record_id IN (...)` 一次回捞，禁止逐条查询 |
| **降级保护** | chunks 为空时自动降级为 `bug_records` 直接检索 |

## 8. 零向量 Parent（Dummy Vector）

**改动**：`import_external.py` 中 parent 记录的 `vector` 字段写入 `[0.0] * dim`

| 考量点 | 决策依据 |
|---|---|
| **检索不再查 parent** | 唯一前门是 `bugvault_chunks`，`bug_records.vector` 永不参与距离计算 |
| **ONNX 推理昂贵** | 2052 条 parent embedding 从 490s 降到 0.1s（5000x 提速） |
| **Schema 合规** | LanceDB 表必须有 vector 字段（列类型限制），零向量满足要求 |

## 9. 时间衰减默认关闭

**改动**：`BUGVAULT_ENABLE_RECENCY_DECAY=false`

| 考量点 | 决策依据 |
|---|---|
| **旧 Bug 仍有价值** | 半年前的排障方案可能仍然正确，时间衰减会错误地压低其排名 |
| **排障场景特殊性** | 不同于新闻/文章检索（时效性重要），bug 修复方案的保质期通常很长 |

## 10. 递归切分（`_split_at_boundary`）

**改动**：`BugRecord.to_chunks()` 中 `body > CHUNK_MAX_SIZE(800)` 时递归按段落/句子边界切分

| 考量点 | 决策依据 |
|---|---|
| **段落优先** | `\n\n` 是自然语义边界，优于固定字数的硬切 |
| **三退级策略** | 段落 → 句子 → 字符，保证总能切成功 |
| **收益边际递减** | 最有价值的是 error_log/semantic 两分刀，内部递归切分只在长文档（>800 字）时有收益 |

## 已否决技术路径汇总

| 否决方案 | 否决理由 |
|---|---|
| LangChain/LangGraph | 线性 CRUD + vector search，框架增加抽象层无收益。BugVault 是工具端点，不是推理引擎 |
| ColBERT 重排 | 需要独立 ~1.5GB PyTorch 索引 + 晚交互存储，5-20 条候选集上 Cross-Encoder 更优 |
| Score Fusion | 向量距离和 BM25 分数量纲不可公度，归一化方法对异常值敏感 |
| Sentence Window Retrieval | 当前数据量下收益有限，`fetch_records_by_ids` 回捞完整 parent 已保证信息完整性 |
| 智能去重 | `fetch_records_by_ids` 天然去重——回捞的是完整 parent，不是 chunk 片段 |
