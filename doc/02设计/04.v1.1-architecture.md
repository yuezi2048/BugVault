# BugVault v1.1 — Architecture Evolution

## Overview

v1.1 在 v1.0 三工具（retrieve / save / reflect）的骨架之上，对**检索链路**和**评估链路**做了全面重构，解决了生产环境中暴露的精度、容灾、路径安全三大问题。

---

## 一、检索链路：从单路到漏斗型

```
v1.0:   Vector ANN → rerank → top-K

v1.1:   Vector ANN (top_k*4) ──┐
        FTS BM25   (top_k*4) ──┤
                                │
        RRF Fusion (k=60) ─────┤
        rerank (时域衰减) ─────┤
        Cross-Encoder 精排 ────┤
        Truncate → top_k ──────┘
```

### 1.1 双路召回

| 路 | 引擎 | 召回数 | 优势 |
|---|---|---|---|
| Dense Vector | fastembed (bge-small-zh-v1.5, 512d) | top_k × 4 | 语义相似度匹配 |
| Full-Text Search | LanceDB Tantivy (BM25) | top_k × 4 | 精确关键词匹配，对特殊符号/变量名友好 |

双路的 `WHERE` 过滤（`tech_stack` / `project_name`）**严格透传**，不会出现一路过滤另一路全表扫描的情况。

### 1.2 RRF 融合

**公式**：`score(d) = Σ 1/(60 + rank_i(d))`

- K=60 业界标准值
- 同时被两路命中的文档，RRF 总分显著高于单路命中的文档
- 不依赖分数量纲（向量距离 vs BM25 无法直接相加）

### 1.3 Cross-Encoder 二次精排

基于 `fastembed.TextCrossEncoder` 的 ONNX 交叉编码器，对 RRF 融合后的 top-K 候选重新打分。

| 模型 | 大小 | 语言 | 默认 |
|---|---|---|---|
| `ms-marco-MiniLM-L-6-v2` | 80MB | 英文 | ✅ 默认 |
| `jina-reranker-v2-base-multilingual` | 1.1GB | 多语言 | 可选 |
| `BAAI/bge-reranker-base` | 1.0GB | 中文 | 可选 |

> **ADR**：为什么放弃 ColBERT 选择 Cross-Encoder？
> 见 [adr-cross-encoder-vs-colbert.md](adr-cross-encoder-vs-colbert.md)

## 二、评估链路：策略模式 + 双重降级

### 2.1 策略模式架构

```
RAGEvalStrategy (Protocol)
  ├── SimpleRAGEvalStrategy       ← v1.0 行为，零变化
  └── ClaimLevelRAGEvalStrategy   ← v1.1 新增：CoT 声明级评估

RAGEvaluator (Facade)
  └── 按 eval_depth 参数运行时选择策略
```

### 2.2 Claim-Level 评估流程

```
用户 query + 检索文档
         │
         ▼
  Step 1: Claim Extraction
  → LLM 从检索文档中提取所有原子事实声明
         │
         ▼
  Step 2: Claim Verification
  → 逐条判断：✅ supported / ❌ unsupported / ⚠️ partial
         │
         ▼
  Step 3: Scoring
  → faithfulness = supported_claims / total_claims
  → context_relevance = 信息需求覆盖度 (0-5)
  → rag_confidence_score = faithfulness × 5 + context_relevance (0-10)
```

### 2.3 双重降级

| 层级 | 触发条件 | 降级目标 | 标记 |
|---|---|---|---|
| 第 1 重 | `max_claim_evals_per_session` 超限 | `SimpleRAGEvalStrategy` | `strategy_used = "simple"` |
| 第 2 重 | 任意异常（JSONDecodeError / 超时 / API 错误） | `SimpleRAGEvalStrategy` | `strategy_used = "simple (fallback_from_error)"` |

> 绝不因评估链路崩溃阻塞检索主线程。

## 三、元数据预过滤

### 参数

| 参数 | 类型 | 作用 |
|---|---|---|
| `target_tech_stack` | `str \| None` | 按技术栈过滤（如 `"Python"`），大小写不敏感 |
| `target_project_name` | `str \| None` | 按项目名过滤（如 `"bugvault-v2"`），大小写不敏感 |

### 安全

`_sanitise_filter_value()` 只允许 `[a-zA-Z0-9_\-. ]`，杜绝 SQL 注入。

### 过滤语法

```sql
LOWER(tech_stack) LIKE '%python%' AND LOWER(project_name) LIKE '%order-svc%'
```

## 四、容灾与高可用

### 4.1 评估链路容灾

```
claim_level.evaluate()
  ├── 成功 → 返回 claim_level 结果
  └── 异常 → logger.warning() → simple.evaluate() → strategy_used = "simple (fallback_from_error)"
```

### 4.2 FTS 引擎容灾

```python
if settings.enable_fts:
    try:
        fts_results = db.search_fts(...)
    except Exception:
        logger.warning("FTS 失败，降级为纯向量检索")
```

### 4.3 路径安全

| 风险 | 修复 |
|---|---|
| `Path("~/.bugvault")` 不展开 | 全局 `.expanduser().resolve()` |
| 写日志前父目录不存在 | `mkdir(parents=True, exist_ok=True)` |
| `scripts/` 直接 `python xxx.py` 报 ModuleNotFoundError | 注释要求必须 `uv run python scripts/xxx.py` |

### 新增配置（v1.1）
