# BugVault v3 架构说明书

> 多智能体系统 + 高级 RAG 架构 + MCP 协议 — Agent 自进化闭环。
> 版本: v3.0 | 最后更新: 2026-05-29

---

## 一、 架构总览

BugVault v3 在 v2 的三大工具基础上，完成了 **P0 数据层鲁棒性加固** 与 **P1/P2 RAG 质量精细化**，形成完整的 Agent 自进化飞轮：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Agent 自进化飞轮                               │
│                                                                     │
│   🛠️ 排障中 ──→  💾 排障后 ──→  🧠 复盘反思 ──→  📚 下次不再犯       │
│   retrieve()      save()         reflect()        CLAUDE.md         │
│        ↑                              ↓                            │
│        └────────── 知识库 ────────────┘                            │
│                  (LanceDB + .md)                                    │
└─────────────────────────────────────────────────────────────────────┘
```

### 三大核心工具

| 阶段 | MCP 工具 | 职责 | 可选 |
|------|----------|------|------|
| 🛠️ 排障中 | `retrieve_bug_experience` | 从历史库检索相似经验，独立精细重排 + RAG 质量评估 | 评估可选 |
| 💾 排障后 | `save_bug_experience` | 零阻塞异步沉淀：Markdown 立即可读 + 后台 Embedding 入库 | 异步可选 |
| 🧠 复盘反思 | `reflect_and_prevent_error` | 分类根因 + 写入 CLAUDE.md，Agent 下次不再犯同样错误 | ✅ 可选 |

### v2 → v3 核心升级

| 维度 | v2 | v3 |
|------|----|----|
| **数据去重** | `_table.add()` 纯追加 | `record_id` MD5 主键 + `merge_insert` 真 Upsert |
| **并发安全** | 无锁，异步线程竞争 | `threading.Lock` + `mode='overwrite'` |
| **相关性底线** | 全量返回 | `MIN_SEMANTIC_SCORE=0.55` 阈值丢弃无关文档 |
| **RAG 评估** | 单一分数 | 三轴 RAGAS：`context_relevance` + `faithfulness` + `justification` |
| **解析健壮性** | 无保护 | `response_format=json_object` + 自动重试 1 次 |

## 二、 三工具职责与内部逻辑

### 2.1 🧠 复盘自纠错工具 — `reflect_and_prevent_error`

**设计理念**：Agent 越用越聪明的核心机制。解决问题后主动反思根因，写入静态上下文（CLAUDE.md），下次启动时自动加载。

```
Agent 请求 ──→ reflect_and_prevent_error
                  │
                  ├─ 读取 CLAUDE.md
                  ├─ 分类根因：
                  │   ├─ understanding_bias   ← 客户隐式需求理解偏差
                  │   ├─ code_logic_error     ← 代码逻辑处理不当
                  │   ├─ api_misuse
                  │   ├─ environment_issue
                  │   └─ other
                  ├─ 追加规则 (递增编号)
                  ├─ 写回 CLAUDE.md
                  └─ 返回元数据 { rule_number, category, total_rules }
```

**CLAUDE.md 格式**：
```markdown
## Bug Prevention Rules

- **Rule #1** [understanding_bias] (2026-05-29T12:00:00+00:00)
  - **Analysis**: 客户未明确说明环境变量配置，误以为是默认值
  - **Prevention**: 修复前先确认所有环境变量，不假设默认值
```

### 2.2 💾 保存工具 — `save_bug_experience`

**设计理念**：零阻塞异步飞轮。SYNC 路径仅做 Markdown 落盘即返回 Agent，ASYNC 钩子在后台静默完成 Embedding + 向量入库。

```
Agent 请求 ──→ save_bug_experience
                  │
                  ├─ SYNC 路径（executor 线程）───── 毫秒级
                  │   ├─ Pydantic 校验 → BugRecord
                  │   │   └─ @model_validator 自动计算 record_id = MD5(title+snippet)
                  │   ├─ validate_and_prepare()
                  │   ├─ archive_svc.write_markdown_archive() → 人类可读 .md
                  │   └─ 立即返回 "saved successfully"
                  │
                  └─ ASYNC 钩子（fire-and-forget）── 后台静默
                      ├─ embedding_svc.generate_embedding()
                      ├─ db.upsert_record()
                      │   └─ merge_insert("record_id")  ← 自动去重
                      │      .when_matched_update_all()
                      │      .when_not_matched_insert_all()
                      └─ 异常仅记录日志，不阻塞 Agent
```

**数据层安全加固（P0）**：

| 防护 | 机制 |
|------|------|
| 重复记录 | `record_id = MD5(bug_title + error_log_snippet)` 主键 |
| 并发读写竞争 | `threading.Lock` 保护 `search()` 和 `upsert_record()` |
| 表版本残留 | `mode='overwrite'` 原子替换，无遗留脏版本引用 |

### 2.3 🛠️ 查询工具 — `retrieve_bug_experience`

**设计理念**：与 Claude 零耦合的独立 Agent 服务。职责专一：ANN 检索 → 精细重排 → 可选 RAG 评估 → 返回格式化数据。Claude 只根据返回结果做决策。

```
Agent 请求 ──→ retrieve_bug_experience
                  │                  ← 以下全部在 BugVault 线程池内完成
                  ├─ embedding_svc.generate_embedding(query)
                  │
                  ├─ db.search(embedding) → Top-K ANN 检索
                  │   └─ threading.Lock 保护（并发安全）
                  │
                  ├─ rerank(results) ← 混合重排
                  │   ├─ semantic = 1.0 - distance/2.0
                  │   ├─ if semantic < MIN_SEMANTIC_SCORE: continue  ← P1 阈值拦截
                  │   ├─ combined = semantic_weight×semantic + recency_weight×recency
                  │   ├─ record_id Set 去重兜底
                  │   └─ 排序后返回
                  │
                  ├─ [可选] rag_evaluator.evaluate_sync()
                  │   ├─ 三轴 LLM 裁判：
                  │   │   ├─ context_relevance (0.0–5.0)
                  │   │   ├─ faithfulness (0.0–5.0)
                  │   │   └─ justification (严厉扣分理由)
                  │   ├─ total = context_relevance + faithfulness
                  │   ├─ 自动重试 1 次（response_format=json_object 防解析崩溃）
                  │   └─ 优雅降级：失败不阻塞检索
                  │
                  └─ 返回给 Agent（RAG 评估作为元数据附加）
                      --- Result 1 ---
                      Title: Connection timeout fix
                      ...
                      --- RAG Evaluation ---
                      Confidence: 8.5/10
                      Assessment: ...
```

**RAG 质量评估输出**（与 Claude 无复杂通信，仅作为额外数据返回）：

```json
{
  "rag_confidence_score": 8.5,     // context_relevance(3.5) + faithfulness(5.0)
  "evaluation": "Context relevance: 3.5 – Doc1 is directly on-topic... Faithfulness: 5.0 – No hallucination detected.",
  "context_relevance": 3.5,
  "faithfulness": 5.0,
  "justification": "Doc2 is partially off-topic..."
}
```

## 四、 数据模型

### 4.1 BugRecord（核心模型）

```python
class BugRecord(BaseModel):
    # ── 必填字段 ──
    bug_title: str              # 简短描述 (1-256 chars)
    error_log_snippet: str      # 错误信息 / 堆栈 (1-32768 chars)
    tried_methods: str          # 尝试路径（含失败） (1-8192 chars)
    final_solution: str         # 最终修复方案 (1-16384 chars)

    # ── 可选字段（支持异步补全）──
    project_name: str | None    # 受影响项目 (≤128)
    tech_stack: str | None      # 技术标签 (≤256)
    root_cause: str | None      # 根因分析 (≤4096)

    # ── 系统管理字段 ──
    record_id: str | None       # MD5(bug_title + error_log_snippet) ← 自动计算
    create_time: str            # ISO-8601 UTC 时间戳（自动生成）

    # ── 内部状态 ──
    _probe_rounds: int = 0      # 追问轮次计数
    MAX_PROBE_ROUNDS: int = 3   # 最大追问轮次
```

**record_id 自动计算**：
```python
@model_validator(mode="after")
def _compute_record_id(self) -> "BugRecord":
    import hashlib
    raw = (self.bug_title + self.error_log_snippet).encode("utf-8")
    self.record_id = hashlib.md5(raw).hexdigest()
    return self
```

### 4.2 RAGEvalResult（三轴评估模型）

```python
class RAGEvalResult(BaseModel):
    rag_confidence_score: float | None  # context_relevance + faithfulness (0–10)
    evaluation: str | None              # justification 的别名（向后兼容）
    context_relevance: float | None     # 0.0–5.0：检索文档的查询有用性
    faithfulness: float | None          # 0.0–5.0：信息对源文档的忠实度
    justification: str | None           # 每条扣分的严厉理由
```

### 4.3 PreventionRule（预防规则模型）

```python
class PreventionRule(BaseModel):
    rule_number: int            # 递增编号
    error_category: str         # 分类：understanding_bias / code_logic_error / ...
    preventive_rule: str        # 预防规则
    reflection_text: str        # 根因分析
    created_at: str             # ISO-8601 时间戳
```

## 六、 P0/P1/P2 修复详解

### 6.1 P0 — 数据重复（哈希主键 + merge_insert）

| 问题 | `_table.add()` 纯追加导致同一 Bug 上报 N 次产生 N 条重复 |
|------|--------------------------------------------------------|
| 修复 | `merge_insert("record_id").when_matched_update_all().when_not_matched_insert_all()` |
| 效果 | 相同 `record_id` 走更新，从源头消除重复 |
| 文件 | `database/lancedb_client.py` `upsert_record()` |

### 6.2 P0 — 并发读写竞争（threading.Lock）

| 问题 | 异步 save 的 embedding + upsert 与 retrieve 的 search 经 2 线程并发时，LanceDB search() 读到旧版本 snapshot 返回 0 条 |
|------|-----------------------------------------------------------------------------------------------------------|
| 修复 | `threading.Lock` 保护 `search()` 和 `upsert_record()` |
| 效果 | 写提交后才被读线程看到，消除竞争条件 |
| 文件 | `database/lancedb_client.py` `__init__()` + `search()` + `upsert_record()` |

### 6.3 P0 — LLM JSON 解析崩溃（response_format + Retry）

| 问题 | LLM 返回 markdown fence 包裹的非标准 JSON 导致 `JSONDecodeError` |
|------|----------------------------------------------------------------|
| 修复 | ① API 强制 `response_format={"type": "json_object"}` ② `evaluate_sync` 重试 1 次 |
| 效果 | JSON 解析 100% 稳定，偶发失败自动重试后不降级 |
| 文件 | `services/rag_evaluator_svc.py` `_call_llm()` + `evaluate_sync()` |

### 6.4 P0 — LanceDB 版本引用残留（mode='overwrite'）

| 问题 | `drop_table` + `create_table(mode='create')` 留下内部旧版本引用 → "file not found" |
|------|-------------------------------------------------------------------------------|
| 修复 | 统一使用 `mode='overwrite'` 原子替换表 |
| 效果 | 无残留脏版本引用，引擎级稳定 |
| 文件 | `database/lancedb_client.py` `_init_table()` |

### 6.5 P1 — 相关性底线阈值

| 问题 | ANN 返回与查询完全无关的文档，污染 TOP-K 结果 |
|------|----------------------------------------------|
| 修复 | `MIN_SEMANTIC_SCORE = 0.55`，低于此值 `continue` 丢弃 |
| 效果 | 宁缺毋滥，无关文档不会出现在最终结果中 |
| 文件 | `services/retrieval_svc.py` `rerank()` |

### 6.6 P2 — 三轴 RAGAS 评估

| 问题 | 单一分数无法区分"检索质量"和"生成忠实度"，分数解释性差 |
|------|------------------------------------------------------|
| 修复 | 分裂为 `context_relevance`(0-5) + `faithfulness`(0-5) + `justification` |
| 效果 | Claude 可明确知道：是检索不准（低 relevance）还是模型编造（低 faithfulness） |
| 文件 | `models/rag_eval_result.py` + `services/rag_evaluator_svc.py` `_parse_response()` |

## 九、 设计原则

### 9.1 Agent 与工具的边界

```
┌─────────────────────┐
│      Agent          │  ← Claude / LLM
│  · 决策             │
│  · 制定修复方案      │
│  · 向用户报告        │
└────────┬────────────┘
         │ 调用 MCP 工具
         ▼
┌─────────────────────┐
│   BugVault Server   │  ← 独立子进程
│  · 检索（ANN + 重排）│
│  · 保存（归档 + 入库）│
│  · 复盘（写入规则）   │
│  · 评估（LLM 裁判）   │
└─────────────────────┘
```

- BugVault **不参与修复决策**，只负责记忆层
- Claude **不直接操作数据库**，只通过 MCP 工具接口交互
- RAG 评估是 **只读指标**，不修改 Claude 的回复（仅附加可信度数据）

### 9.2 自进化闭环

```
排障请求
   │
   ▼
┌──────────────┐     ┌───────────────────┐
│  检索历史经验 ◄────┤ BugVault 语义检索  │ ← ANN + 精细重排 + 阈值
│  (Agent 决策) │     │  + RAG 评估       │
└──────┬───────┘     └───────────────────┘
       │
       ▼
┌──────────────┐     ┌───────────────────┐
│  生成修复方案 │     │                   │
│  (Agent)     │     │   Agent 自行决策   │
└──────┬───────┘     │   采纳/忽略评估    │
       │             │                   │
       ▼             └───────────────────┘
┌──────────────┐     ┌───────────────────┐
│  保存经验     ◄────┤ BugVault 异步入库  │ ← MD 归档 + Embedding
│  (Agent 主动) │     │  + merge_insert    │
└──────┬───────┘     └───────────────────┘
       │
       ▼
┌──────────────┐     ┌───────────────────┐
│  复盘反思     ◄────┤ BugVault 写入      │ ← CLAUDE.md 预防规则
│  (Agent 主动) │     │  CLAUDE.md        │
└──────┬───────┘     └───────────────────┘
       │
       ▼
   下次排障 ──→ 知识库更丰富 + 预防规则已加载
```

## 十、 安全边界与容错机制

| 场景 | 保护 | 效果 |
|------|------|------|
| Embedding 模型下载失败 | `unset all_proxy` / 缓存 `~/.cache/fastembed/` | 离线可运行 |
| LanceDB 表版本残留 | `mode='overwrite'` | 启动无报错 |
